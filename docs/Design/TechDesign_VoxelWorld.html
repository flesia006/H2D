1. 목적과 범위

Minecraft 식 블록 기반 변경 가능 월드를 Unreal Engine 5에서 구현하고, 시드(seed) 값으로 결정되는 절차적 생성으로 무한한 지형을 생성하는 시스템의 구현 스펙.

범위 안

  • 고정 크기 청크의 동적 로딩·언로딩
  • 블록 단위 채굴·배치
  • 시드 기반 결정론적 생성 (높이맵 · 바이옴 · 동굴 · 광물 · 구조물)
  • 청크 메시 생성 (페이스 컬링 · 선택적 그리디 메싱)
  • 세이브: 청크 델타만 저장

범위 밖 (별도 문서)

  • 전투 / 인카운터 시스템 → TechDesign_GAS.md (예정)
  • 크래프팅 · 하우징 UI → TechDesign_Crafting.md (예정)
  • HD-2D 셰이딩 / 포스트프로세스 디테일 → VisualTone.md (예정)

2. 핵심 설계 결정

항목결정근거
청크 크기 16 × 16 × 16 (cube) 3차원 동굴/구조물에 유리. 메시 갱신·세이브 단위 균등. Minecraft(16×16×256)는 평면 게임용.
블록 단위 길이 100 cm (1 UE 유닛) UE 기본 좌표계와 일치. 픽셀 캐릭터 키 약 2~3블록.
렌더링 ProceduralMeshComponent (DIY) 학습 목적. 대안: RealtimeMeshComponent(플러그인), Voxel Plugin(상용).
구현 언어 C++ 모듈 + Blueprint 노출 청크 메시·노이즈는 성능상 C++ 필수. 게임 로직은 BP 친화.
노이즈 라이브러리 FastNoise2 (또는 자체 Perlin) 오픈소스, UE 통합 가능. 학습용으로 자체 Perlin 구현도 추천.
스레딩 FNonAbandonableTask 비동기 청크 생성 게임 스레드 hitch 방지. 메시 적용만 게임 스레드.
월드 한계 안전 거점 + 야생 영역 구분 (Concept B 결정) 1인 개발 안전장치. 자유 건축은 거점 한정.

3. 데이터 모델

3.1 블록 (Block)

한 블록은 1바이트 ID로 인코딩 (최대 256종). 속성은 별도 테이블에서 ID로 조회.

using BlockId = uint8;

enum class EBlockId : BlockId {
    Air = 0,
    Stone, Dirt, Grass, Sand, Water, Wood, Leaves,
    CoalOre, IronOre, GoldOre,
    Workbench, Bed,            // 기능 블록
    // ...
    Count
};

USTRUCT()
struct FBlockType {
    GENERATED_BODY()
    FString  Name;
    bool     bSolid       = true;   // 충돌·메싱 대상
    bool     bTransparent = false;  // 메시 페이스 컬링 판정
    float    Hardness     = 1.0f;   // 채굴 시간
    int32    AtlasIndex[6];         // ±X ±Y ±Z 면별 텍스처 인덱스
    EBlockId DropId       = EBlockId::Air;
};

블록 타입 테이블은 UDataTable로 관리해 BP에서 편집 가능하게 한다.

3.2 청크 (Chunk)

16³ = 4096 블록. ID 1바이트 × 4096 = 4 KB raw.

constexpr int32 CHUNK_SIZE = 16;
constexpr int32 CHUNK_VOLUME = CHUNK_SIZE * CHUNK_SIZE * CHUNK_SIZE;

struct FChunk {
    FIntVector Coord;                       // 청크 좌표 (월드 좌표 = Coord * 16)
    TArray<BlockId> Blocks;                 // size = CHUNK_VOLUME, index = X + Y*16 + Z*256
    bool bDirty = false;                    // 메시 재생성 필요?
    bool bModified = false;                 // 세이브 대상 (사용자 수정)
    TArray<FBlockEdit> Edits;               // 절차적 생성 결과와의 델타

    BlockId Get(int32 X, int32 Y, int32 Z) const {
        return Blocks[X + Y*CHUNK_SIZE + Z*CHUNK_SIZE*CHUNK_SIZE];
    }
    void Set(int32 X, int32 Y, int32 Z, BlockId Id) {
        Blocks[X + Y*CHUNK_SIZE + Z*CHUNK_SIZE*CHUNK_SIZE] = Id;
        bDirty = true; bModified = true;
    }
};

struct FBlockEdit { uint16 LocalIndex; BlockId NewId; };

플레이어 위치 기준 로딩 반경 R 안의 청크를 우선순위 큐로 로드. 반경 R+버퍼 밖 청크는 언로드.

4. 절차적 생성 파이프라인

4.1 시드와 결정론적 RNG

같은 시드 + 같은 좌표는 항상 같은 결과를 내야 한다 (재현성).

// 좌표별 결정론적 시드 파생
uint64 DeriveSeed(int64 WorldSeed, int32 X, int32 Y, int32 Z, uint32 LayerSalt) {
    uint64 H = (uint64)WorldSeed;
    H ^= HashCombineFast(X, 0x9E3779B97F4A7C15ull);
    H ^= HashCombineFast(Y, 0xBF58476D1CE4E5B9ull);
    H ^= HashCombineFast(Z, 0x94D049BB133111EBull);
    H ^= LayerSalt * 0xD2B74407B1CE6E93ull;
    return H;
}

각 노이즈 레이어는 고유 LayerSalt 값을 사용 → 레이어 간 패턴 독립.

4.2 노이즈 레이어

4.2.1 높이맵 (2D)

// (worldX, worldY) → 지표면 높이 Z
float HeightAt(int32 wx, int32 wy) {
    float continent = Perlin2D(wx * 0.001f, wy * 0.001f, SaltContinent); // 대륙 (저주파)
    float hill      = Perlin2D(wx * 0.01f,  wy * 0.01f,  SaltHill) * 0.5f;
    float detail    = Perlin2D(wx * 0.05f,  wy * 0.05f,  SaltDetail) * 0.1f;
    float h01 = (continent + hill + detail) * 0.5f + 0.5f; // 0~1
    return FMath::Lerp(MinHeight, MaxHeight, h01);         // 예: 32~96
}

4.2.2 바이옴 (온도 + 습도)

온도 \ 습도건조중간습함
차가움툰드라침엽수림설원 늪
온화평원숲늪
더움사막사바나정글

온도/습도 각각 저주파 Perlin 2D. 바이옴은 표면 블록(잔디/모래/눈) + 식생 + 광물 분포에 영향.

4.2.3 동굴 (3D)

3D 노이즈 임계값으로 빈 공간을 깎아낸다.

// Z < surface 인 블록에 대해
float n = Perlin3D(wx*0.04f, wy*0.04f, wz*0.04f, SaltCave);
if (n > CaveThreshold) chunk.Set(.., EBlockId::Air);  // 빈 공간

worm-cave(긴 통로)는 1.5단계: 시작점에서 Bezier·랜덤 워크로 채굴 — Phase 4에서 선택 도입.

4.2.4 광물

각 광물은 (레이어, 빈도, 클러스터 크기, Z 범위)로 정의. Z가 낮을수록 희귀 광물.

FOreVein veins[] = {
  { EBlockId::CoalOre,  freq=0.04, cluster=6, zRange=[ 0, 64] },
  { EBlockId::IronOre,  freq=0.02, cluster=5, zRange=[ 0, 40] },
  { EBlockId::GoldOre,  freq=0.005,cluster=4, zRange=[ 0, 20] },
};

4.2.5 구조물 (마을 · 작은 던전)

2단 결정: (1) 청크 별 구조물 존재 여부를 시드 해시로 결정. (2) 존재 시 템플릿 적용.

if (HashChunk(seed, chunkCoord) % 1000 < 5) {  // 0.5% 청크에 마을
    SpawnStructure(EStructure::Village, chunkCoord);
}

구조물 템플릿은 NBT 식 정형 데이터 (블록 + 오프셋 배열) 또는 UDataTable로 관리.

4.3 생성 순서 (파이프라인)

  1. Allocate — 청크 4096 블록을 Air로 채움
  2. Terrain — 높이맵 + 표면 블록 배치 (Stone / Dirt / Grass)
  3. Biome Paint — 표면 블록을 바이옴별로 치환 (Sand / Snow / ...)
  4. Caves — 3D 노이즈로 Air 카빙
  5. Ores — 광물 베인 배치 (Stone만 치환)
  6. Structures — 마을·던전 템플릿 스탬핑 (청크 경계 가능)
  7. Decoration — 풀·꽃·나무 등 식생
  8. Apply Edits — 세이브된 사용자 델타를 마지막에 덮어쓰기
구조물·식생은 청크 경계를 넘을 수 있다. 이웃 청크에 미리 "pending edits"를 큐잉해두는 패턴으로 해결.

5. 청크 메싱

5.1 페이스 컬링 (naive)

각 솔리드 블록의 6개 면 중, 이웃이 투명(Air/Water/Leaves)인 면만 메시에 추가.

for (int z=0; z<16; ++z) for (int y=0; y<16; ++y) for (int x=0; x<16; ++x) {
    if (!IsSolid(chunk.Get(x,y,z))) continue;
    for (int face=0; face<6; ++face) {
        FIntVector n = NeighborOffset[face];
        if (IsTransparent(WorldGet(x+n.X, y+n.Y, z+n.Z)))
            EmitQuad(x, y, z, face);
    }
}

16³ 청크에서 평균 1k~3k 쿼드. Phase 1~2는 이걸로 충분.

5.2 그리디 메싱 (선택)

같은 텍스처·면 방향의 인접 쿼드를 큰 사각형으로 병합 → 정점 수 1/5~1/10. 구현 비용은 있음.

Phase 4 이후 성능 병목이 보이면 도입. 그 전엔 시간 낭비.

5.3 텍스처 아틀라스 + UV

모든 블록 면 텍스처를 하나의 아틀라스(예: 16×16 grid, 각 칸 32×32 px)에 모은다.

  • 드로 콜 1개로 청크 전체 렌더 가능
  • UV는 FBlockType.AtlasIndex[face] 로 계산
  • 밉맵 사용 시 가장자리 블리딩 방지를 위해 각 칸에 1~2 px 패딩

6. UE5 통합

6.1 ProceduralMeshComponent

청크별 1개 UProceduralMeshComponent. 메시는 단일 섹션, 단일 머티리얼.

UCLASS()
class AChunkActor : public AActor {
    GENERATED_BODY()
    UPROPERTY(VisibleAnywhere)
    UProceduralMeshComponent* Mesh;
    FIntVector Coord;
    void BuildMesh(const FChunkMeshData& Data);  // game-thread 적용
};

6.2 비동기 생성

// 워커 스레드: 생성 + 메싱
class FChunkGenTask : public FNonAbandonableTask {
public:
    FIntVector Coord; int64 Seed;
    FChunkMeshData Result;
    void DoWork() {
        FChunk Chunk = GenerateChunk(Seed, Coord);  // 4.3 파이프라인
        Result = BuildMesh(Chunk);                  // 5.1
    }
    TStatId GetStatId() const { RETURN_QUICK_DECLARE_CYCLE_STAT(FChunkGenTask, STATGROUP_ThreadPoolAsyncTasks); }
};

// 게임 스레드: 결과 수신 후 AChunkActor에 적용
AsyncTask(ENamedThreads::GameThread, [Coord, Result]() {
    ApplyChunkMesh(Coord, Result);
});

6.3 머티리얼

  • 마스터 머티리얼 1개 (Unlit 또는 Lit + 단순 토너) + 아틀라스 텍스처
  • 특수 블록(물·잎)은 별도 머티리얼 / 별도 메시 섹션
  • 버텍스 컬러로 바이옴 틴팅 (잔디 색상 등)

6.4 충돌

두 가지 옵션:

  • A. 메시 자체에 Collision 생성 — CreateMeshSection_LinearColor(..., bCreateCollision=true). 간단하지만 갱신 비용 있음.
  • B. 별도 Collision 청크 — 솔리드 박스 그룹으로 단순화 (그리디 메싱 결과 재활용 가능)

Phase 1~3은 A로, Phase 4+ 병목 시 B로 전환.

6.5 청크 LRU 캐시

로딩 반경 밖 청크는 즉시 파괴하지 않고 N분 LRU 캐시에 보존 (왕복 이동 시 재생성 비용 절감).

7. 세이브 / 로드

7.1 청크 델타

절차적 생성은 시드+좌표로 재현 가능하므로, 사용자가 수정한 블록만 저장.

FArchive& operator<<(FArchive& Ar, FChunkSaveData& D) {
    Ar << D.Coord;
    int32 Count = D.Edits.Num();
    Ar << Count;
    for (int i=0; i<Count; ++i) {
        Ar << D.Edits[i].LocalIndex;
        Ar << D.Edits[i].NewId;
    }
    return Ar;
}

1 edit = 3 byte. 청크당 평균 수십~수백 edit 가정 → 청크당 수백 byte 수준.

7.2 월드 메타

  • WorldSeed (int64)
  • 월드 버전 (마이그레이션 용)
  • 플레이어 위치 · 인벤토리 · 거점 좌표

전체 파일은 USaveGame 기반. 거대 월드는 청크별 파일 분리 (Saved/SaveGames/<world>/chunks/<x_y_z>.bin).

8. HD-2D 톤 통합 (참조)

본 시스템은 지오메트리만 담당. 시각 톤(HD-2D 매핑)은 다음으로 분담:

  • 마스터 머티리얼: 약한 라이팅, 페인팅 톤 컬러 그루핑, 소프트 섀도우
  • 포스트프로세스 볼륨: 블룸 · 뎁스 페이드 · 약한 필름 그레인
  • 카메라: 3/4 시점, 약간의 perspective (orthographic 아님). 큐브가 비스듬히 보이게.
  • 픽셀 캐릭터: PaperZD 기존 자산. 라이팅에 반응하도록 정점 컬러 또는 라이트 펑션 적용.

상세는 별도 VisualTone.md 에서 다룬다.

10. 단계별 구현 계획

Phase 1 — 단일 청크 큐브 메시 (1~2주)

  • C++ 모듈 HybridVoxel 추가, UProceduralMeshComponent 의존성
  • FChunk, BlockId, FBlockType 데이터 정의
  • 고정 데이터로 16³ 청크 채우고 페이스 컬링 메시 생성
  • AChunkActor + 단일 머티리얼 + 임시 텍스처
  • 완료 기준: 에디터에서 한 청크가 정상 렌더, 충돌 동작

Phase 2 — 다중 청크 동적 로딩 (1~2주)

  • UVoxelWorld + 로드/언로드 큐
  • 플레이어 위치 기준 반경 로딩
  • FChunkGenTask 비동기화
  • 채굴·배치 입력 (좌클릭/우클릭)
  • 인접 청크 경계 메시 정합성 (이웃 블록 조회)
  • 완료 기준: 끝없는 평지에서 이동·채굴·배치 동작

Phase 3 — 노이즈 기반 지형 (2주)

  • 시드·좌표 파생 RNG
  • 높이맵 노이즈 (대륙 + 언덕 + 디테일)
  • 표면 블록 배치 (Stone/Dirt/Grass)
  • 텍스처 아틀라스 + UV 인덱싱
  • 완료 기준: 한 시드에서 자연스러운 언덕·평원 생성. 같은 시드 = 같은 지형 재현 확인.

Phase 4 — 바이옴 + 동굴 + 광물 (2~3주)

  • 온도·습도 2D 노이즈 → 바이옴 ID
  • 표면 블록 바이옴별 치환 (모래·눈·잔디)
  • 3D 노이즈 카브로 동굴 생성
  • 광물 베인 (Coal/Iron/Gold, Z 범위)
  • 풀·나무 식생 (이웃 청크 pending edits 포함)
  • 완료 기준: 바이옴 전환 자연스럽고, 동굴 탐험·광물 채집 가능

Phase 5 — 구조물 + 세이브 (2주)
