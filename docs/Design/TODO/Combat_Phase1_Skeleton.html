Combat Phase 1 — JRPG 전투 골격 (1대1)

예상 1주 · Task 7개 · 목표: 인카운터 → 턴 진행 → 승/패/도망까지 완결
← 인덱스 · 톤: 옥토파스 트래블러·드래곤 퀘스트 라인의 클래식 JRPG

설계 의도와 범위

이번 Phase는 전투 시스템의 골격을 1대1 최소 구성으로 끝낸다. 데이터 모델·상태머신·액션 해결·UI 계약·인카운터 전환의 전체 경로를 통과시키는 것이 목표이며, 파티(다인)·스킬·아이템·속성·상태이상은 Phase 2 이후로 미룬다. 이렇게 하면 추가 기능이 모두 동일한 골격 위에 얹히는 형태가 되어 확장이 단순해진다.

아키텍처: 커스텀 시스템(데이터 자산 기반). GAS는 채택하지 않는다 — 턴제는 GAS의 실시간 모델과 잘 안 맞아 어댑터 비용이 크고, 클래식 JRPG 흐름은 명시적 상태머신이 디버깅하기 더 쉽다.

완료 시 모습

  • 맵 위 인카운터 트리거(임의 액터/볼륨)가 닿으면 전투 매니저가 시작되고 입력이 전투 UI로 넘어간다.
  • 속도(SPD) 순으로 차례가 돌고, 플레이어 차례엔 액션 선택(공격/방어/도망), 적 차례엔 단순 AI가 자동 선택한다.
  • HP가 0이 된 쪽이 패배. 적 패배 시 보상(임시: 로그 메시지), 플레이어 패배 시 게임오버 페이드.
  • 도망 성공 시 즉시 전투 종료, 실패 시 차례 소진.
  • 전투 매니저는 BP에서 호출 가능한 인터페이스(BeginCombat, SubmitPlayerAction 등)를 제공한다.

상태: TODO

목적

전투 중 가변 상태(현재 HP/MP, 일시 버프)는 정의(Def)와 분리해 런타임 인스턴스로 들고 있는다.

코드

USTRUCT(BlueprintType)
struct FTPCombatant {
    GENERATED_BODY()
    UPROPERTY() TObjectPtr<const UTPCombatantDef> Def = nullptr;

    UPROPERTY(BlueprintReadOnly) int32 CurHP = 0;
    UPROPERTY(BlueprintReadOnly) int32 CurMP = 0;
    UPROPERTY(BlueprintReadOnly) bool  bDefending = false; // 이번 턴 방어 중
    UPROPERTY(BlueprintReadOnly) bool  bDead = false;

    void InitFrom(const UTPCombatantDef* InDef) {
        Def = InDef; CurHP = InDef->Stats.MaxHP; CurMP = InDef->Stats.MaxMP;
        bDefending = false; bDead = false;
    }
    int32 SPD() const { return Def ? Def->Stats.SPD : 0; }
    bool  IsPlayer() const { return Def && Def->bIsPlayerControlled; }
};

Task 1.5 — 액션 해결 & 데미지 공식

상태: TODO

목적

고전 JRPG식 단순 공식. "튠은 데이터로, 분기는 코드로"가 원칙.

코드

// 기본 공격
static int32 ComputeAttackDamage(const FTPCombatant& A, const FTPCombatant& D, float Power)
{
    const int32 Atk = A.Def->Stats.ATK;
    const int32 Def = D.bDefending ? (D.Def->Stats.DEF * 3 / 2) : D.Def->Stats.DEF;
    int32 Dmg = FMath::Max(1, FMath::RoundToInt(Atk * Power) - Def);
    // ±10% 변동 + 크리 (LUK% 확률, 1.5배)
    Dmg = FMath::RoundToInt(Dmg * FMath::FRandRange(0.9f, 1.1f));
    if (FMath::FRand() < A.Def->Stats.LUK / 100.f) Dmg = Dmg * 3 / 2;
    return FMath::Max(1, Dmg);
}

// 도망 성공률: 적과의 SPD 차에 LUK 가산
static bool RollFlee(const FTPCombatant& Me, const FTPCombatant& Foe)
{
    const float Base = 0.5f + 0.05f * (Me.SPD() - Foe.SPD()) + Me.Def->Stats.LUK * 0.01f;
    return FMath::FRand() < FMath::Clamp(Base, 0.1f, 0.95f);
}

방어는 1턴 동안 bDefending=true를 켜 두고 라운드 끝에 리셋. 도망 실패는 차례를 그냥 소진한다.

DoD · 동일 인풋에서 데미지 분포가 합리적 범위(약 ±10% + 크리)이고, 방어 시 받는 피해가 명확히 감소한다.

Task 1.6 — UI / 입력 계약 (BP 측)

상태: TODO

목적

전투 매니저는 데이터·로직만 책임지고, 표시·입력은 UMG가 담당. 두 영역의 인터페이스를 명확히 굳혀 둔다.

BP가 매니저에 의존하는 것

  • BeginCombat(PlayerDef, EnemyDef) 호출
  • SubmitPlayerAction(ActionDef, TargetIndex) 호출
  • OnPhaseChanged(NewPhase), OnCombatLog(FString) 구독
  • BP에서 읽을 상태: GetPlayer(), GetEnemy(), GetCurrentTurnIndex()

위젯 구성 (WBP_Combat)

  • 적/플레이어 스프라이트 슬롯 2개, HP 바
  • 액션 메뉴 (3 버튼: 공격/방어/도망), 대상이 둘 이상일 땐 대상 선택 패널(Phase 2 대비)
  • 로그 텍스트 영역(이벤트 누적, 3~5줄)
  • OnPhaseChanged == AwaitInput 일 때만 입력 활성화
DoD · UI 없이 콘솔 로그만으로도 매니저가 단독 동작(테스트 가능), UI를 붙이면 동일 흐름이 시각화됨.
